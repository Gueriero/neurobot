# Дизайн: бэкфилл заказов на 30 дней + колонка "Остаток"

Дата: 2026-05-15

Две независимые фичи на листе "Заказы и Остатки":

- **Фича A** — одноразовый бэкфилл исторических заказов на 30 дней (отдельный скрипт).
- **Фича B** — ежедневная колонка F "Остаток" с общим остатком по артикулу.

---

## Фича B: колонка F "Остаток" (ежедневная, ongoing)

### Что

Новая meta-колонка `F "Остаток"` после `E "Склад"`. Константа `META_COLS` меняется `5 → 6`.

- **Строка артикула** (пустой Склад) → `F` = сумма по всем складам этого `nm_id`:
  `quantity + in_way_to + in_way_from`. То же число, что в колонке "Всего остатков"
  листа "Себестоимость всех остатков".
- **Строки складов** → `F` пусто.
- Значение перезаписывается каждый запуск. История не хранится (одна ячейка на артикул).

### Где код

`sheets_service.update_combined_sheet`:
- Тоталы считаются из уже передаваемого аргумента `stocks_data` — новый параметр
  не нужен. Агрегация: для каждого `(ip, nm_id)` суммируем `quantity + in_way_to + in_way_from`
  по всем складам.
- Правятся все места, завязанные на `META_COLS`: срезы `headers[5:]` / `row[META_COLS:]`,
  заголовки строк (`new_headers`, summary-строки), A1-диапазоны записи, индексы групп строк,
  диапазон basic filter, индекс колонки даты для заметок.

`main.py` — изменений практически нет (`stocks_data` уже формируется и передаётся).

### Миграция старого листа

Лист сейчас имеет 5 meta-колонок. Первый запуск с `META_COLS = 6`:
- метод читает лист, проверяет `headers[5]`;
- если `headers[5] != 'Остаток'` → лист в старом layout: трактуем индекс 5 как первую
  колонку даты на чтении, затем пишем уже новый layout с колонкой F;
- если `headers[5] == 'Остаток'` → лист уже мигрирован, читаем как 6 meta-колонок.

Ручных правок листа не требуется.

## Фича A: бэкфилл 30 дней заказов (one-off, отдельный скрипт)

Полностью отдельный скрипт `src/backfill_orders.py`. Основной поток (`main.py`,
дневной сбор) не затрагивает. Возможность документируется в `CLAUDE.md`.

### Проблема текущего скрипта

`src/backfill_orders.py` сейчас вызывает `OrdersClient.BASE_URL`, который указывает на
`/api/analytics/v3/sales-funnel/products/**history**` — эндпоинт с максимумом в одну
неделю и без `cancelCount`/`cancelSum`. Использовать его для 30-дневного бэкфилла нельзя.
Скрипт переписывается на `/api/analytics/v3/sales-funnel/products` (агрегат за период;
`start == end` → суточная нарезка, глубина до 365 дней).

### Логика

1. Прочитать самую раннюю колонку-дату с листа "Заказы и Остатки".
2. Диапазон бэкфилла: `[earliest - 30 дней ... earliest - 1 день]`.
3. `nmIds` — все активные артикулы из таблицы `articles` (per IP).
4. Для каждого IP, для каждого дня окна:
   - `POST https://seller-analytics-api.wildberries.ru/api/analytics/v3/sales-funnel/products`
   - заголовки: `Authorization: <WB_API_KEY_*>`, `Content-Type: application/json`
     (тот же ключ, что используется для Stocks/Orders сейчас)
   - body: `{"selectedPeriod": {"start": day, "end": day}, "nmIds": chunk, "limit": 1000}`
   - `nmIds` чанкуется по ≤1000 (`limit` обязателен, иначе WB обрежет ответ до 50 карточек).
5. Парсинг ответа: `data.products[i].product.nmId` + `statistic.selected.orderCount`
   (и `orderSum`, `cancelCount` — для возможного будущего использования; на лист идёт
   `orderCount`). Пустой `products: []` на HTTP 200 — норма (нет трафика), не ошибка.

### Реализация — sync

Кодовая база синхронная (`requests`), а не `asyncio`/`httpx`. Скрипт пишется
синхронно по образцу из хендоффа, адаптированному под `requests` + `time.sleep`:

- **Глобальный пейсер**: `≥ 20 c` между ЛЮБЫМИ запросами к funnel (process-wide,
  не per-account) — последовательный обход кабинетов иначе сложит запросы вплотную.
- **Retry**: 5 попыток, backoff `30/60/120/240/480 c` на `429` и `5xx` и таймаутах.
- **Non-retryable**: `400/401/403` → исключение `FunnelApiError` (не молчать,
  иначе фейл выглядит как success).
- **Прерывание кабинета**: при `FunnelApiError` или исчерпании retry окно для этого
  IP прерывается целиком — не «грызём» остаток дней зарейтлимиченного кабинета.

### Запись в лист — bulk

Текущий подход (вызвать `update_combined_sheet` по разу на каждую из 30 дат) — это
~180 вызовов Sheets API и 30× clear/rewrite всего листа. Вместо этого новый метод:

`sheets_service.update_combined_sheet_bulk(orders_by_date: dict)`
- `orders_by_date`: `{ "DD.MM.YYYY": { (ip, nm_id, title): orders_count, ... }, ... }`
- один `get_all_values` → мерж всех дат в `rows_map` → одна запись → один regroup.
- ~6 вызовов Sheets API вместо ~180. Безопаснее (нет 30 промежуточных состояний листа).
- Учитывает `META_COLS = 6` (колонка F), как и `update_combined_sheet`.

### В БД не пишем

Бэкфилл пишет **только в лист**. Таблица `orders_raw` не трогается.

### Объём

При <1000 nmIds на IP — 1 запрос/день/IP. 30 дней × 3 ИП ≈ 90 запросов × 20 c ≈ 30 минут.
Одноразовый прогон.

## Порядок реализации

1. **Фича B** первой — меняет `META_COLS` и layout листа.
2. **Фича A** второй — её код записи в лист (`update_combined_sheet_bulk`) должен
   совпадать с новым layout (6 meta-колонок).
3. Документация возможности бэкфилла в `CLAUDE.md`.

## Файлы

- `src/sheets_service.py` — `META_COLS` 5→6, правки `update_combined_sheet`,
  миграция layout, новый `update_combined_sheet_bulk`.
- `src/backfill_orders.py` — полная переписка на `/sales-funnel/products` day-by-day,
  sync-пейсер, bulk-запись в лист, без записи в БД.
- `src/main.py` — без изменений (или минимальная сверка передачи `stocks_data`).
- `CLAUDE.md` — секция про бэкфилл.

## Тестирование

- Фича B: прогон дневного сбора на тестовом/боевом листе — проверить, что колонка F
  появилась, на строках артикулов стоит сумма остатков, на строках складов пусто,
  группировка/фильтр/сортировка не сломаны; повторный прогон не дублирует колонку.
- Фича A: `--dry-run` показывает корректное окно дат; реальный прогон на 2-3 днях
  (урезанное окно) — проверить появление исторических колонок и значений заказов
  на пересечении с артикулом.
